# Epitaxy v0.2 — Systems-Engineer Dashboard

Closes the gap from v0.1 (structural data layer + minimal drill-down) to v0.2 (queryable + silent-failure surface). Filter: prioritize features that make CACE / system-level failure modes visible; explicitly defer pedagogical-textbook polish to the v2/v3 stretch (HTML Explainer Generator, see [ROADMAP §5](../ROADMAP.md#5-stretch-v2v3--html-explainer-generator)).

This is also Epitaxy's first `plan` node — eating its own dog food per CLAUDE.md core principle 2.

## 0. Why this spec exists

v0.1 shipped 2026-05-17 (PR4 `9508651`). On 2026-05-19 dogfood — adding ADR-011 (timezone normalization) + POR YAML on 3 RecSys scripts — surfaced two findings:

1. **One real v0 schema gap**: POR `decisions:` is stored as data but emits **no graph edge**. ADR-011 lives in the index but `module:src/.../run_features.py` is 0 edges away from it. See §3.1.
2. **One UX framing mistake**: `epi serve` currently renders like a directory listing. The temptation is to bolt on pedagogical polish (diagrams, reading paths, glossary popovers) — but the [ML systems failure-mode citations](../../README.md) thesis says that's the wrong audience. v0.2's user is the MLE mid-iteration, not the new-reader. See §2.

This doc fixes the scope before v0.2 implementation begins, mirroring the design-first pattern that made PR1-4 work.

## 1. Position relative to v0.1

| Component                 | v0.1 status                          | v0.2 delta                                                                                       |
| ------------------------- | ------------------------------------ | ------------------------------------------------------------------------------------------------ |
| `epi sync` schema         | 5 node types / 4 edge types          | + `follows` edge (module/function → adr, source: `por-frontmatter`)                              |
| `epi sync` parser         | POR YAML parsed into `node.por` dict | Final-pass emits `follows` edges from `por.decisions[]`; dangling targets preserved per SCHEMA §6 |
| `epi serve` UI            | Module-centric drill-down            | + Impact preview panel + parameter lineage + provenance badges + drift indicators                |

| ROADMAP positioning       | v0 → v1 (Pillar 1 Bootstrap)         | v0 → **v0.2** → v1 — adds a polish milestone before the pillar-1 commitment                      |

No version bump for v0.1.x point releases; v0.2 is the next breaking-scope release. Pydantic models will gain `follows` enum entry — that's the only schema change.

## 2. Thesis-derived priority filter

The [`reference_ml_systems_failure_modes`](file:///home/baron/.claude/projects/-home-baron-PycharmProjects-Epitaxy/memory/reference_ml_systems_failure_modes.md) memory collects the academic + industry citations (Sculley 2015 CACE, Huyen "ML systems often fail silently," Shankar 2023 Berkeley, Lietzau Scrum critique, MBSE) that ground Epitaxy's product framing. The crisp filter it derives:

> **`epi serve` should look like a systems-engineer dashboard, not a pedagogical textbook.** ML/AI failure modes are silent and system-level; the user mid-iteration converts on "what changes if I touch this" — not on reading-paths or hover-glossary polish.

Apply this filter to every candidate v0.2 feature:

| Skip (UX candy, no thesis leverage)              | Prioritize (makes silent failure visible)                                  |
| ------------------------------------------------ | -------------------------------------------------------------------------- |
| Glossary hover popovers (CSS-only or JS)         | Impact preview ("if I change this module, here's what it touches")         |
| Reading-paths sidebar (3 min / 15 min / 45 min)  | Parameter lineage visualization (ADR → ParameterNode → consuming functions) |
| SVG pedagogical cards (rank-1 / rank-2 imagery)  | Drift indicators (data layer vs code layer inconsistencies)                |
| Q&A pressure-test format                         | Provenance badges (`🤖 ast` / `📝 frontmatter+body` / `👤 manual`)          |
| Multi-audience reading framing                   | POR `why:` assumption surfacing (highlights "we assume X" sentences)       |
| Status chips on banner (Offline ✓ / Pending)     | "Outdated since" timestamps when an ADR's referenced module was last edited |
| Mermaid diagram rendering in markdown body       | Cross-file consistency checks (e.g., same parameter mentioned in 2 POR blocks → flag if values differ) |

The right column is what differentiates Epitaxy from "yet another doc explorer". The left column is what `phase2_new_released.html`-style hand-written pedagogical docs already do well; LLM-compose narrative replication of that is the v2/v3 stretch (ROADMAP §5), not v0.2.

## 3. Scope — what ships in v0.2

### 3.1 `follows` edge type (mandatory; first sub-PR)

**Problem captured during 2026-05-19 dogfood**: a module with POR YAML

```yaml
decisions:
  - adr:docs/decisions/011-timezone-normalization-rule.md
```

correctly parses into `node.por.decisions` as data, but the graph has **0 edges** from that module to ADR-011. Downstream `por_trace` / `por_lineage` MCP tools can read the field directly, but the drill-down UI has no edge to render, so the module detail page shows the ADR ID as plain text — not navigable.

**Spec**:

- Add `follows` to [docs/SCHEMA.md §3](../SCHEMA.md) edge-type enum
- Direction: `from = module|function`, `to = adr`
- Source label: `por-frontmatter`
- Provenance: `por-frontmatter` (mirrors `adr-frontmatter` for `decides` edges)
- Emission: final pass after POR parse (mirrors the `populate_decided_by` pass added in PR4)
- Dangling-target rule: per SCHEMA §6 amendment, emit edge even when target ADR is absent (drift signal). Apply same rule that `supersedes` and `decides` use.
- Tests:
  1. Module with POR `decisions: [adr:...existing.md]` → emit 1 `follows` edge
  2. Function-level POR docstring with `decisions:` → emit `follows` from function ID
  3. POR `decisions:` references missing ADR ID → dangling `follows` edge (warn but don't fail)
  4. POR YAML present but no `decisions:` field → no `follows` edges
  5. Multiple `decisions:` entries → one edge each
  6. Backward compat: existing PR2 lock-guard tests for the missing edge type flip to assert presence

Effort: 1-2 days. Single PR (`v0.2-PR1`).

### 3.2 Impact preview UI (highest-leverage feature)

**Thesis target**: directly answers "if I change X, what else changes?" — the CACE-visibility ask.

**Spec**: on every module / function / parameter detail page, render an "Impact preview" panel above the existing POR section:

```
┌─ Impact preview ──────────────────────────────────┐
│ Changing this module would affect:                │
│                                                   │
│ ▸ 3 downstream consumers (depends-on):            │
│   • module:src/.../compose.py                     │
│   • module:src/.../run_evaluation.py              │
│   • function:src/.../main()                       │
│                                                   │
│ ▸ 2 ADRs whose decisions involve this code:       │
│   • adr:docs/decisions/003-...md                  │
│   • adr:docs/decisions/011-timezone-...md         │
│                                                   │
│ ▸ 1 parameter decided in this scope:              │
│   • param:.../EWR_THRESHOLD_LONG                  │
└───────────────────────────────────────────────────┘
```

Edge types to traverse:

- **`depends-on` reverse** (who calls me) — already in v0.1 index
- **`follows` reverse** (which ADRs cite me as decided code) — added in §3.1
- **`decides` from same scope** — for parameters scoped to this module
- **`references` reverse** (ADR/plan body links to me) — already in v0.1 index

Limit BFS depth to 1 hop in v0.2 (one level of impact). Multi-hop (transitive impact) is v0.3+.

Effort: 3-4 days. PR `v0.2-PR2`. Depends on §3.1.

### 3.3 Parameter lineage visualization

**Thesis target**: train-serving skew detection — when the same parameter lives in two `# epitaxy:param` markers, surface it.

**Spec**: parameter detail page shows the full chain:

```
ADR-005 (decides) ──▶ EWR_THRESHOLD_LONG ──▶ used in: [
                                              function:.../compute_features
                                              function:.../run_evaluation
                                            ]
```

Function detail page lists parameters it consumes (via existing `depends-on` to parameter scope).

**Drift sub-feature** (v0.2-PR3 sub-task): when two `parameter` nodes share the same `name` but live in different `scope`, render a yellow warning on both detail pages: *"Same parameter name in 2 scopes — verify these are intentionally separate."* This catches train-serving skew at the index level, not just at MCP query time.

Effort: 2 days. PR `v0.2-PR3`. Depends on §3.2 (shares UI infra).

### 3.4 Provenance badges

**Already specified in [ROADMAP §4](../ROADMAP.md#4-design-principle--progressive-enhancement)**; not yet implemented.

**Spec**: every node and edge in serve UI shows provenance:

- `🤖 ast` (Python AST-extracted, no human signal)
- `📝 frontmatter+body` (markdown frontmatter parsed)
- `🔗 por-frontmatter` (POR YAML in docstring)
- `👤 manual` (reserved for v1+ when humans override)

CSS-only badge rendering — no JS island, no extra payload. Inline next to each rendered field.

Effort: 1 day. Bundled into PR `v0.2-PR3`.

### 3.5 Drift indicators (preview only; full detection is v2b)

**Spec**: surface inconsistencies the parser *already detects* but currently swallows or shows as warnings only:

- POR `decisions:` references a nonexistent ADR ID → red "dangling" badge on the link
- POR `prereqs:` mentions a table / file / module that isn't in the graph → yellow "unverified prereq" badge
- ADR `status: superseded` but no `supersedes:` target field → red "broken supersession chain" badge
- ADR last modified before a referenced module's last commit → yellow "potentially stale" badge (requires git log read at sync time — opt-in via `--check-staleness` flag)

This is **not** the full v2b drift detection (that needs an event-driven daemon comparing live state). v0.2 only renders signals that exist at sync time.

Effort: 1-2 days. PR `v0.2-PR4`.

## 4. Explicit non-goals (deferred to v2/v3 stretch)

Per the thesis filter, **NOT** in v0.2:

- Mermaid / KaTeX / SVG rendering in markdown body
- Reading-paths sidebar
- Glossary hover popovers
- Q&A format rendering
- Multi-audience framing
- Status chips on banner
- LLM-compose narrative segments (this is ROADMAP §5 stretch — gated on real OSS traction, never v0.x)
- Pillar 1 Bootstrap (`epi bootstrap`) — v1 commitment
- Pillar 2a in-session MCP `prompts/` exports — v1 commitment

Lock-in: if a future session is tempted to add any of these to v0.2 scope, the answer is "no — this is v0.2 polish gate, see ROADMAP filter / [reference_ml_systems_failure_modes](file:///home/baron/.claude/projects/-home-baron-PycharmProjects-Epitaxy/memory/reference_ml_systems_failure_modes.md) memory". Revisit only if HN response (post-v0.2) reveals demand specifically for pedagogical-doc replication.

## 6. Exit criteria for v0.2 ship

- All 4 sub-PRs merged to `main`
- Test coverage maintained ≥ 90% (PR4 baseline)
- `epi sync` regression check: re-run on Epitaxy itself + RecSys; stats numbers within ±2 of pre-v0.2 baseline for non-edge counts; edge count strictly grows (new `follows` edges only add)
- [docs/SCHEMA.md](../SCHEMA.md) amended: §3 adds `follows` to edge enum; §6 extends dangling-target rule to `follows`
- [docs/ROADMAP.md](../ROADMAP.md) §3 phasing table: insert v0.2 row between v0 and v1
- Memory: add `feedback_v02_skip_pedagogical_polish.md` to lock the filter for future sessions (prevents scope drift in v0.3+)
- Tag `v0.2.0`. Classifier still `2 - Pre-Alpha`.
